# 매주 기도문 PDF 2개 제작·등록 규칙

## 실행 기준

- 매주 화요일 23:00 Asia/Seoul에 서로 다른 신규 주제의 7일 유료 기도문 PDF를 정확히 2개 제작한다.
- 매 실행을 시작할 때 해당 주차의 제작·검수·관리자 등록을 완료하는 goal을 생성한다.
- 결과는 `D:\codex\sinang\prayPDF\주간생성\YYYY-MM-DD\상품명\`에 저장한다.
- 상품 가격과 추가 인쇄 이용권 금액은 웹 상품·결제 정보에서만 관리한다. 유료본, 미리보기, 썸네일, 업로드 문구에는 가격을 쓰지 않는다.
- 기도문 PDF 페이지에 직접 필요한 사이트 코드, 결제, DB, Git 커밋·배포, 관리자 자료 등록은 별도 승인 없이 진행한다. 관련 없는 사이트 영역은 변경하지 않는다.

## 중복 방지

1. `data/prayer-pdf-registry.json`에서 기존 상품 제목, 상품 주제, 일차 주제, 장절, 핵심 초점, 실천 키를 확인한다.
2. 두 신규 상품은 기존 자료 및 서로 간에 의미상 중복되지 않아야 한다. 동의어로 바꾼 재사용도 중복이다.
3. 자동 검증은 정확히 같은 값의 재사용을 차단하며, 의미상 중복은 생성 전에 원고 전체를 대조한다.
4. 두 상품의 시각 검수까지 통과한 뒤에만 `--register-only`로 장부에 등록한다.
5. 첫 활성 실행의 첫 상품은 `자녀를 위한 7일 잠언 기도문`으로 한다. 이후에는 장부에 등록된 주제를 다시 쓰지 않는다.

## 상품 원고 규격

- `tools/generate_prayer_product_from_spec_v3.py`가 읽는 UTF-8 JSON 원고를 상품별로 만든다.
- 원고에는 상품별 `prayer_goal`과 선택 가능한 `cover.style`을 넣는다. 표지는 주제별 원본 벡터 모티프로 다르게 구성하되 시리즈 정체성은 유지한다.
- 각 7일 상품은 표지, 자료 정보, 사용 안내, 7일 읽기·기록 페이지쌍, 함께 활용하기 2쪽, 여정 회고, 이용 안내의 21쪽이다.
- 공개 미리보기는 표지, 자료 정보, 1일차 읽기, 1일차 기록의 정확히 4쪽이다.
- 자료 정보에는 `자료 제목`, `짧은 요약`, `키워드`, `상세 설명`을 모두 표시한다.
- 업로드 문구는 `자료 제목`, `짧은 요약`, `상세 설명`, `키워드`, `5. 공개자료 구성` 순서다.
- `5. 공개자료 구성`에는 전체 21쪽 구성, 7일 전체 목차, 기도 목표, 공개 미리보기 범위를 명시한다.
- 각 일차는 고유한 `topic`, `reference`, `focus`, `practice_key`, 개역한글 본문 1~3절, 묵상, 직접 읽는 기도문, 기록 질문, 작은 실천을 가진다.
- 개역한글 본문은 `D:\codex\tmp\krv-source-full\json\88.json`과 문장 단위로 일치해야 한다.
- 구매 1건은 개인·가정 이용과 교회·기관 비영리 인쇄 20부까지를 포함한다. 초과 부수 가격은 PDF가 아니라 웹 주문에서만 계산한다.

## 실행과 검수

1. 두 JSON 원고를 `data/product_specs/YYYY-MM-DD/`에 저장한다.
2. 두 원고를 한 명령에 전달해 교차 중복과 기존 장부 중복을 검사한다.
3. 유료본 21쪽, 미리보기 4쪽, A4 규격을 텍스트 추출과 PDF 구조로 확인한다.
4. PDF, 미리보기, 썸네일, 업로드 문구에서 `판매가`, 원화 가격, 통화 기호가 없는지 확인한다.
5. 업로드 문구에 총 21쪽 구성, 7개 일차 목차, `prayer_goal`, 4쪽 미리보기 범위가 모두 있는지 확인한다.
6. 144 dpi PNG로 표지, 자료 정보, 가장 긴 읽기, 기록, 함께 활용하기 2쪽, 이용 안내를 육안 검수한다.
7. 두 상품 모두 통과하면 `--register-only`를 실행해 장부에 추가한다.
8. 날짜 폴더에 `생성_검수_보고서.md`를 작성한다.
9. 관리자 계정으로 `https://product-codex-90j.pages.dev/admin-faith-resources`에 접속해 PDF 원본, 4쪽 미리보기, 업로드 문구와 상품 정보를 등록하고 공개한다. 기존 상품 개선은 동일 ID를 수정한다.
10. 인증 세션이 없거나 관리자 등록이 실패하면 접근 제어를 우회하지 않는다. 검증 파일을 보존하고 등록 대기 상태와 정확한 원인을 보고한다.
