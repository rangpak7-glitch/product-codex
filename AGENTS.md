# AGENTS.md

## 프로젝트 개요

이 프로젝트는 `기도의샘물` 웹사이트다.

`기도의샘물`은 기독교인의 실제 기도 생활을 돕기 위한 말씀·기도 콘텐츠 공간이다. 단순 소개형 홈페이지가 아니라, 매일의 말씀과 기도문, 아침기도, 저녁기도, 영상묵상, 말씀·기도 카드, 30일 기도 여정, 프리미엄 PDF 자료가 서로 연결되는 신앙 콘텐츠 허브로 발전시키는 것을 목표로 한다.

현재는 정적 HTML/CSS/JavaScript 구조이지만, 향후 React 등 동적 프레임워크로 전환될 수 있다. 따라서 새 기능은 가능하면 데이터와 렌더링 구조를 분리하고, HTML에 콘텐츠를 과도하게 직접 박아 넣지 않는다.

## 핵심 브랜드 방향

- 브랜드명은 항상 `기도의샘물`로 표기한다.
- 사이트는 "기도와 말씀으로 쉬어가는 곳"이라는 인상을 주어야 한다.
- 주요 방문자는 40~70대 성도까지 포함하므로 글씨 크기, 줄간격, 버튼 터치 영역, 대비를 넉넉하게 잡는다.
- 젊은 사용자도 촌스럽게 느끼지 않도록 여백, 카드 구조, 색상, 아이콘을 정돈된 프리미엄 톤으로 유지한다.
- 분위기는 조용함, 따뜻함, 평안함, 신뢰감, 프리미엄을 기준으로 한다.
- 과장된 치유 약속, 즉시 해결, 응답 보장 같은 표현은 쓰지 않는다.

## 유튜브 채널 연계 기준

연계 채널:

`https://youtube.com/channel/UCFrsilNKJ8xcmn0RUrFz6XQ?si=hM09ebhKwa9m_3W0`

`기도의샘물` 웹사이트는 유튜브 채널의 보조 링크 모음이 아니라, 영상 콘텐츠를 말씀·기도 아카이브와 유료 자료 흐름으로 확장하는 중심 허브가 되어야 한다.

### 채널 역할 분석

- 유튜브는 사용자가 기도와 말씀을 "듣는" 입구다.
- 웹사이트는 사용자가 기도와 말씀을 "읽고, 다시 찾고, 저장하고, 나누는" 공간이다.
- 유튜브 영상은 무료 위로 콘텐츠로 유지하고, 웹사이트는 날짜별 아카이브, 카테고리별 기도문, 말씀 카드, PDF 자료, 30일 기도 여정으로 깊이를 더한다.
- 영상 시청 후 웹사이트로 들어온 사용자가 오늘의 말씀, 관련 기도문, 지난 묵상, 신앙자료를 자연스럽게 찾도록 연결한다.
- 웹사이트에서 유튜브로 나가는 링크는 단순 채널 홍보가 아니라 해당 페이지 주제와 관련된 영상묵상으로 이어지는 흐름이어야 한다.

### 채널-웹사이트 콘텐츠 흐름

기본 흐름:

1. 유튜브에서 무료 영상묵상 또는 기도 영상을 듣는다.
2. 웹사이트에서 같은 주제의 말씀과 기도문을 읽는다.
3. 날짜별 아카이브에서 지난 말씀을 다시 찾는다.
4. 말씀·기도 카드, 30일 기도 여정, 프리미엄 PDF 같은 신앙자료로 확장한다.
5. 필요하면 기도제목을 남기거나 문의한다.

페이지별 연결 기준:

- `index.html`: 오늘의 말씀, 오늘의 기도, 최근 아카이브, 유튜브 채널 CTA를 함께 보여주는 메인 허브다.
- `videos.html`: 실제 유튜브 영상 카드와 썸네일을 통해 채널로 이동하는 영상묵상 허브다.
- `night-prayer.html`: 잠들기 직전 보는 밤기도와 수면 전 영상묵상을 연결한다.
- `morning-prayer.html`: 하루를 여는 아침기도와 아침 말씀 영상을 연결한다.
- `meditation.html`: 성경 이야기와 오늘의 삶을 연결하는 신앙묵상 페이지다. 영상묵상 자체는 `videos.html`에서 담당한다.
- `prayer-cards.html`: 말씀을 이미지 카드로 저장하거나 나누고 싶은 사용자에게 연결한다.
- `premium-pdf.html`: 광고 없이 조용히 읽고 반복할 수 있는 PDF 자료로 연결한다.
- `archive.html`: 날짜별·카테고리별 지난 말씀을 다시 찾는 중심 페이지다.

### 유튜브 데이터 사용 원칙

- 영상 링크는 가능한 실제 영상 단위 URL을 사용한다.
- 채널 URL만 반복하지 말고, 영상 제목, 주제, 관련 말씀, 설명, 썸네일을 데이터로 관리한다.
- YouTube 공식 썸네일 URL 패턴은 영상 이동 목적의 카드에서만 사용한다.
- 무단 복제 이미지, 로고, 타 사이트 자산을 가져오지 않는다.
- 유튜브 구독 유도는 가능하지만 과도한 반복, 클릭 강요, 광고 클릭 유도 문구는 금지한다.
- 영상묵상 CTA는 "유튜브에서 먼저 듣기", "영상묵상 보기"처럼 자연스러운 문구를 사용한다.

## 콘텐츠 구조

주요 콘텐츠 축:

- 오늘 붙드는 말씀
- 말씀 붙들기
- 아침기도
- 저녁기도
- 영상묵상
- 신앙자료
- 기도제목
- 문의하기
- 지난 말씀 아카이브

데이터 중심 파일:

- `data/dailyContents.js`: 말씀 붙들기, 저녁기도, 아침기도, 신앙묵상의 날짜별 일일 콘텐츠
- `data/prayers.js`: 상황별 기도문
- `data/meditations.js`: 성경 말씀 묵상
- `data/videos.js`: 유튜브 영상묵상
- `data/prayerCards.js`: 말씀·기도 카드
- `data/prayerChallenge.js`: 30일 기도 여정
- `data/pdfProducts.js`: 프리미엄 PDF 상품
- `data/dailyWords.js`: 기존 홈/아카이브용 날짜별 오늘의 말씀과 기도. 신규 카테고리 페이지는 `data/dailyContents.js`를 우선한다.
- `data/categories.js`: 카테고리 메타데이터

날짜형 콘텐츠는 한국 시간 `Asia/Seoul` 기준으로 생각한다. 오늘 날짜 콘텐츠가 없으면 가장 최근 콘텐츠를 보여주는 fallback을 유지한다.

### 일일 콘텐츠 카테고리 체계

신규 일일 콘텐츠 카테고리는 다음 4개를 기준으로 한다.

- `word` / 말씀 붙들기 / `prayers.html`: 오늘 하루 마음에 붙들 한 구절, 한 문장 묵상, 고백, 짧은 기도
- `evening` / 저녁기도 / `night-prayer.html`: 하루 정리, 내려놓음, 잠들기 전 평안, 평안 고백문
- `morning` / 아침기도 / `morning-prayer.html`: 감사, 지혜, 말과 생각, 오늘 실천할 한 가지
- `editorial` / 신앙묵상 / `meditation.html`: 일반 기도문이 아니라 성경 이야기와 오늘의 삶을 연결하는 사설형 묵상

표시 기준:

- `Asia/Seoul` 기준 오전 7시 이전에는 전날 콘텐츠를 표시한다.
- 오전 7시 이후에는 오늘 날짜 콘텐츠를 표시한다.
- 해당 날짜 콘텐츠가 없으면 같은 카테고리의 가장 최근 콘텐츠를 표시한다.
- 각 카테고리 페이지에는 오늘 콘텐츠, 달력 아카이브, 최근 7일/30일 보기를 우선 제공한다.

`data/dailyContents.js` 표준 필드:

- `id`, `date`, `category`, `categoryLabel`, `title`
- `scriptureRef`, `scriptureText`, `summary`, `body`
- `prayer`, `confession`
- `editorialQuestion`, `editorialInsight`, `application`
- `tags`, `imageTheme`, `icon`, `detailUrl`

영상묵상은 일일 콘텐츠 4대 카테고리와 분리해 `videos.html`과 `data/videos.js`에서 관리한다. `data/videos.js`에는 실제 `videoId`, 제목, 주제, 관련 말씀, 설명, 태그, 게시일을 넣고, 썸네일은 YouTube 공식 URL 패턴 `https://i.ytimg.com/vi/{videoId}/hqdefault.jpg`를 사용한다.

## 작성 톤

- 문장은 차분하고 따뜻하게 쓴다.
- 신앙적 위로는 제공하되, 문제 해결이나 치유를 보장하지 않는다.
- "반드시 응답받는 기도", "기도만 하면 치유됩니다", "즉시 해결", "구매하면 문제가 해결됩니다" 같은 표현은 금지한다.
- "말씀을 붙드는 시간", "하나님 앞에 마음을 내려놓는 기도", "천천히 읽고 묵상할 수 있는 자료" 같은 표현을 우선한다.
- 성경 본문은 길게 대량 인용하지 말고, 짧은 구절과 출처 중심으로 사용한다.

- 텍스트 메인: `#17213B`

- 포인트 골드: `#C7A15A`
- 세이지 그린: `#A8B8A0`

디자인 원칙:

- 실제 이미지가 없을 때는 CSS gradient, radial light, paper texture, abstract shape, inline SVG icon으로 완성도 있게 표현한다.
- 외부 사이트 이미지, Pinterest 이미지, sarang.org 이미지, 타 사이트 로고나 아이콘을 복사하지 않는다.
- Pinterest와 참고 사이트는 분위기, 색상, 구도 참고용으로만 사용한다.
- 카드 UI는 얇은 테두리, 부드러운 그림자, 카테고리 라벨, SVG 아이콘, 충분한 여백을 기본으로 한다.
- 모바일에서는 카드와 CTA가 1열로 자연스럽게 정렬되어야 한다.
- 텍스트 줄바꿈은 `word-break: keep-all`, 넉넉한 `line-height`, 적절한 `max-width`를 고려한다.

## 동적 UI 기준

- 기도문 카드와 말씀·기도 카드는 단순 그리드보다 3D 스택, 캐러셀, 좌우 이동 UI를 우선 고려한다.
- 좌우 버튼, 키보드 방향키, 모바일 스와이프를 지원하면 좋다.
- 애니메이션은 차분하고 절제되어야 하며, 기도문 읽기를 방해하면 안 된다.
- 검색, 필터, 아카이브 기능은 데이터 기반으로 구현한다.
- 향후 React 전환을 고려해 렌더링 함수와 데이터 구조를 분리한다.

## 수익화 흐름

초기 수익화 우선순위:

1. 기도문 PDF 패키지
2. 광고 없는 오디오 묵상 또는 오디오북
3. 30일 기도 챌린지
4. 말씀·기도 카드 이미지팩
5. AI 맞춤 기도문 생성

표현 기준:

- 유료자료는 무료 콘텐츠를 대체하는 것이 아니라 "더 오래 곁에 두고 읽고 싶은 사람을 위한 확장 자료"로 설명한다.
- 초기 버튼 문구는 `구매하기`보다 `자료 문의하기`, `신앙자료 둘러보기`, `30일 기도 여정 보기`, `유튜브에서 먼저 듣기`를 우선한다.
- 결제 기능은 향후 확장 주석으로 남길 수 있다.

## SEO / AdSense 기준

- 새 HTML 페이지에는 UTF-8, viewport, description, canonical, OG title/description/url/image, title, AdSense account meta, CSS 링크를 포함한다.
- 광고 영역이 본문보다 먼저 보이거나 콘텐츠 경험을 방해하지 않게 한다.
- 광고 클릭 유도 문구는 금지한다.
- 새 페이지를 만들면 `sitemap.xml` 반영 여부를 체크한다.
- 정책 페이지 링크는 유지한다: `privacy.html`, `terms.html`, `disclaimer.html`, `sitemap.xml`.

## 작업 안전 규칙

- 작업 전 `git status --short`로 현재 변경 사항을 확인한다.
- 사용자가 허용한 파일 외에는 수정하지 않는다.
- 기존 콘텐츠, 메뉴 링크, SEO meta, 정책 페이지 링크, AdSense 주석, 유튜브 링크, 문의 링크를 임의로 삭제하거나 변경하지 않는다.
- 디자인 개선이 목적이어도 HTML 구조를 과도하게 갈아엎지 말고 필요한 섹션 단위로 수정한다.
- 새 파일이 필요하면 이유를 설명하고 허용된 경우에만 만든다.
- 문제가 생기면 되돌릴 수 있도록 백업을 먼저 만든다.
- `generate_site.py`는 현재 HTML 인코딩을 깨뜨릴 위험이 있으므로 명시 요청 없이 실행하지 않는다.
- 루트의 `style.css`, `script.js`보다 실제 로드되는 `assets/css/style.css`, `assets/js/script.js`를 우선 확인한다.

## 검증 체크리스트

작업 후 최소한 다음을 확인한다.

- `git status --short`
- `git diff --check`
- JS 파일 변경 시 `node --check`
- 로컬 preview 서버에서 주요 페이지 HTTP 200 확인
- 모바일 폭에서 카드, 버튼, 메뉴가 깨지지 않는지 확인
- 사용자 화면에 `샘플`, `임시`, `준비 중`, `TODO`, `MVP` 같은 표현이 노출되지 않는지 확인
- 유튜브 링크, 문의 링크, 정책 링크가 유지되는지 확인
- 새 페이지가 있으면 `sitemap.xml` 반영 필요 여부를 보고한다.

## 배포 기준

- 배포 전 커밋 범위를 명확히 확인한다.
- 백업 폴더는 일반적으로 배포 커밋에 포함하지 않는다.
- 사용자가 preview 검토 후 배포를 요청하면, 검증을 다시 수행하고 의도한 파일만 커밋한다.
- `git push` 후 원격 브랜치와 커밋 해시를 보고한다.
