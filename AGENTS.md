## 최우선 작업 범위 보호 규칙

이 규칙은 모든 분석, 수정, 검수 작업에 우선 적용한다. 사용자가 현재 작업에서 명시적으로 허용한 범위만 예외로 하며, 명시되지 않은 변경은 허용된 것으로 간주하지 않는다. 아래 규칙과 이후 지침이 충돌하면 이 섹션을 따른다.

### 1. 프로젝트 핵심 파일 이해

- `index.html`은 홈페이지 진입 파일이다.
- 공통 디자인과 동작은 `assets/css/style.css`, `assets/css/editorial-redesign.css`, `assets/js/script.js`, `assets/js/faith-member.js`, `data/*.js`에 분산되어 있을 수 있으므로, 수정 전 연결 관계를 확인한다.
- 로그인, 회원, 구매, 결제, 다운로드 관련 파일과 `workers/**`는 별도 승인 없이는 수정하지 않는 민감 영역이다.
- 특정 파일만 원본으로 간주해 인라인 CSS, 인라인 JavaScript, 중복 구현을 추가하지 않는다.

### 2. 읽기 전용 기본값

- 사용자가 `분석`, `검토`, `평가`, `계획`, `제안`만 요청하면 파일을 생성·수정·삭제하지 않는다.
- 실제 수정은 사용자가 `수정해줘`, `구현해줘`, `적용해줘`처럼 명시적으로 승인한 뒤에만 한다.
- 계획 승인과 구현 승인은 별개의 단계다.

### 3. 수정 전 필수 확인과 보고

실제 수정 전에 반드시 다음을 수행한다.

1. `git status --short`로 현재 변경 사항을 확인한다.
2. `git diff --name-only`와 필요한 대상 파일의 `git diff -- <파일>`로 기존 미커밋 변경을 확인한다.
3. 사용자에게 수정 예정 파일, 각 파일의 목적, 수정할 섹션 또는 기능, 기존 기능에 미칠 영향, 새 파일 필요 여부를 먼저 보고한다.
4. 사용자가 허용한 파일 목록을 확정한 뒤에만 수정한다.

작업 대상 파일에 사용자의 기존 미커밋 변경이 있으면 그 내용을 덮어쓰거나 임의로 병합하지 말고 먼저 보고하여 승인받는다.

### 4. 허용 범위 외 변경 금지

- 사용자가 허용한 파일 외에는 수정·생성·삭제·이름 변경·이동하지 않는다.
- 추가 파일이 필요하면 이유, 변경하지 않을 경우의 영향, 예상 변경 범위를 먼저 보고하고 별도 승인을 받는다.
- 더 나은 구조, 코드 정리, 일관성, 최적화, 베스트 프랙티스를 이유로 범위 밖 리팩터링을 하지 않는다.
- 요청과 직접 관련 없는 오류는 수정하지 말고 별도로 보고한다.

### 5. 최소 변경 원칙

- 디자인 개선이라도 HTML 구조를 전면 재작성하지 않고 필요한 요소와 섹션만 수정한다.
- 기존 선택자, `id`, `class`, `data-*`, 폼 필드 이름과 JavaScript 연결 관계를 먼저 확인한다.
- JavaScript가 참조하는 식별자를 임의로 삭제하거나 변경하지 않는다.
- 파일 전체 재생성, 저장소 전체 검색·치환, 관련 없는 포맷팅·공백·주석·변수명 변경을 하지 않는다.
- UTF-8 인코딩과 기존 줄바꿈 형식을 유지한다.

### 6. 반드시 보존할 내용

별도 승인 없이 다음을 삭제하거나 변경하지 않는다.

- 기존 콘텐츠와 콘텐츠 데이터, 카테고리 종류·명칭·순서, 메뉴 링크와 기존 페이지 URL
- 쿼리 파라미터와 콘텐츠 ID, SEO 메타(`title`, `description`, `canonical`, Open Graph)
- Google Analytics, Microsoft Clarity, AdSense 메타·스크립트·주석·광고 위치
- 유튜브 채널·영상 링크, 문의 링크와 문의 폼 action
- 개인정보처리방침·이용약관·콘텐츠 안내 링크와 본문
- 로그인·회원가입·비밀번호 재설정, 게시판 글·답글·수정·삭제 기능
- Supabase 테이블·컬럼·RLS 정책·저장 데이터, 상품·주문·구매 권한·결제·다운로드 기능
- Cloudflare Worker 설정·비밀값, `robots.txt`, `sitemap.xml`, 운영 도메인 `https://product-codex-90j.pages.dev/`

### 7. 금지된 Git 및 파일 작업

사용자의 명시적 승인 없이 다음을 하지 않는다.

- `git reset --hard`, `git clean`, `git restore`, `git checkout --`, `git stash`
- 기존 변경 사항 되돌리기, 알 수 없는 파일 또는 사용자 파일 삭제, 브랜치 변경
- commit, push, merge, force push, 배포, 태그 생성

사용자의 기존 변경을 불필요하다고 판단해 제거하지 않는다.

### 8. 새 파일·의존성·도구

- 새 파일은 생성 전 이유, 위치, 용도, 배포 영향을 설명하고 승인을 받는다.
- 승인 없이 패키지·라이브러리·프레임워크·폰트·이미지·빌드 도구를 설치하거나 다운로드하지 않는다.
- 승인 없이 `package.json`, lock 파일, 배포 설정, GitHub Actions, Cloudflare 설정을 변경하지 않는다.
- `generate_site.py`는 명시적 승인 없이는 실행하거나 수정하지 않는다.
- 임시 파일은 저장소 밖에서 사용하고 작업 뒤 제거한다.
- 저장소 안에 `.bak`, `backup/`, 복제 HTML 등 배포될 수 있는 백업 파일을 만들지 않는다.

### 9. 복구 가능성

- 기존 Git 기록을 기본 복구 수단으로 사용한다.
- 작업 대상에 미커밋 변경이 있어 별도 백업이 필요하면 먼저 보고한다.
- 승인된 경우에만 저장소 밖에 diff patch를 보관한다.
- 백업을 이유로 임의 commit 또는 stash를 하지 않는다.
- 변경은 작은 단위의 patch로 적용해 쉽게 되돌릴 수 있게 한다.

### 10. 작업 후 필수 검증

수정 후 최소한 다음을 확인한다.

1. `git status --short`, `git diff --check`, `git diff --stat`
2. 허용된 파일 외 변경 여부
3. JavaScript 수정 시 해당 파일의 `node --check`
4. 주요 페이지의 로컬 HTTP 응답, 데스크톱·모바일 동작, 브라우저 콘솔 오류
5. 영향받은 메뉴·링크·로그인·폼·검색·필터 기능
6. SEO 메타, 정책 링크, 유튜브 링크, 문의 링크 보존 여부
7. `robots.txt`와 `sitemap.xml`이 허용 없이 변경되지 않았는지 여부

테스트 과정에서 파일을 만들면 자신이 생성한 파일만 제거하며 기존 파일은 삭제하지 않는다.

### 11. 완료 보고 기준

전체 파일 내용을 출력하지 않고 다음만 간결히 보고한다.

1. 실제 수정 파일 목록과 파일별 요약
2. 요청 범위 밖 파일 미변경 확인
3. 검증 명령과 결과, 정상 확인한 기능
4. 남은 위험 또는 확인 필요 사항과 되돌리는 방법
5. commit, push, 배포를 수행하지 않았다는 확인

## 프로젝트 개요

이 프로젝트는 `기도의샘물` 웹사이트다.

`기도의샘물`은 기독교인의 실제 기도 생활을 돕기 위한 말씀·기도 콘텐츠 공간이다. 단순 소개형 홈페이지가 아니라, 매일의 말씀과 기도문, 아침기도, 저녁기도, 기도(유튜브), 말씀·기도 카드, 30일 기도 여정, 프리미엄 PDF 자료가 서로 연결되는 신앙 콘텐츠 허브로 발전시키는 것을 목표로 한다.

현재는 정적 HTML/CSS/JavaScript 구조이지만, 향후 React 등 동적 프레임워크로 전환될 수 있다. 따라서 새 기능은 가능하면 데이터와 렌더링 구조를 분리하고, HTML에 콘텐츠를 과도하게 직접 박아 넣지 않는다.

## 핵심 브랜드 방향

- 브랜드명은 항상 `기도의샘물`로 표기한다.
- 사이트는 "기도와 말씀으로 쉬어가는 곳"이라는 인상을 주어야 한다.
- 주요 방문자는 40~70대 성도까지 포함하므로 글씨 크기, 줄간격, 버튼 터치 영역, 대비를 넉넉하게 잡는다.
- 젊은 사용자도 촌스럽게 느끼지 않도록 여백, 카드 구조, 색상, 아이콘을 정돈된 프리미엄 톤으로 유지한다.
- 분위기는 조용함, 따뜻함, 평안함, 신뢰감, 프리미엄을 기준으로 한다.
- 과장된 치유 약속, 즉시 해결, 응답 보장 같은 표현은 쓰지 않는다.

- 유튜브 구독 유도는 가능하지만 과도한 반복, 클릭 강요, 광고 클릭 유도 문구는 금지한다.

### 일일 콘텐츠 카테고리 체계

신규 일일 콘텐츠 카테고리는 다음 4개를 기준으로 한다.

- `Asia/Seoul` 기준 오전 6시 이전에는 전날 콘텐츠를 표시한다.
- 오전 6시 이후에는 오늘 날짜 콘텐츠를 표시한다.
- 해당 날짜 콘텐츠가 없으면 같은 카테고리의 가장 최근 콘텐츠를 표시한다.
- 각 카테고리 페이지에는 오늘 콘텐츠, 달력 아카이브, 최근 7일/30일 보기를 우선 제공한다.

`data/dailyContents.js` 표준 필드:

- `id`, `date`, `category`, `categoryLabel`, `title`
- `scriptureRef`, `scriptureText`, `summary`, `body`
- `prayer`, `confession`
- `editorialQuestion`, `editorialInsight`, `application`
- `tags`, `imageTheme`, `icon`, `detailUrl`

### 일일 콘텐츠 자동화

- GitHub Actions는 한국 시간 오전 6시(UTC `21:00`)에 `data/dailyContents.js`의 4개 일일 카테고리를 생성한다.
- 자동화는 GitHub Secret `OPENAI_API_KEY`와 GitHub Variable `OPENAI_MODEL`만 사용한다. 키 값은 저장소, 로그, 커밋에 넣지 않는다.
- 같은 날짜와 카테고리의 기존 항목은 보존하며, 4개 항목이 모두 검증될 때만 파일을 갱신한다.
- 생성 실패 시 기존 콘텐츠를 유지하고 커밋하지 않는다.

기도(유튜브)은 일일 콘텐츠 4대 카테고리와 분리해 `videos.html`과 `data/videos.js`에서 관리한다. `data/videos.js`에는 실제 `videoId`, 제목, 주제, 관련 말씀, 설명, 태그, 게시일을 넣고, 썸네일은 YouTube 공식 URL 패턴 `https://i.ytimg.com/vi/{videoId}/hqdefault.jpg`를 사용한다.

## 작성 톤

- 문장은 차분하고 따뜻하게 쓴다.
- 신앙적 위로는 제공하되, 문제 해결이나 치유를 보장하지 않는다.
- "반드시 응답받는 기도", "기도만 하면 치유됩니다", "즉시 해결", "구매하면 문제가 해결됩니다" 같은 표현은 금지한다.
- "말씀을 붙드는 시간", "하나님 앞에 마음을 내려놓는 기도", "천천히 읽고 묵상할 수 있는 자료" 같은 표현을 우선한다.
- 성경 본문은 길게 대량 인용하지 말고, 짧은 구절과 출처 중심으로 사용한다.

## 디자인 기준

색상:

- 메인 배경: `#FAF7F0`, `#F8F5EF`
- 보조 배경: `#EFE7DA`
- 텍스트 메인: `#17213B`
- 텍스트 보조: `#6E7688`
- 딥 네이비: `#111B33`
- 포인트 골드: `#C7A15A`
- 세이지 그린: `#A8B8A0`

디자인 원칙:

- 실제 이미지가 없을 때는 CSS gradient, radial light, paper texture, abstract shape, inline SVG icon으로 완성도 있게 표현한다.
- 외부 사이트 이미지, Pinterest 이미지, sarang.org 이미지, 타 사이트 로고나 아이콘을 복사하지 않는다.
- Pinterest와 참고 사이트는 분위기, 색상, 구도 참고용으로만 사용한다.
- 카드 UI는 얇은 테두리, 부드러운 그림자, 카테고리 라벨, SVG 아이콘, 충분한 여백을 기본으로 한다.
- 모바일에서는 카드와 CTA가 1열로 자연스럽게 정렬되어야 한다.
- 텍스트 줄바꿈은 `word-break: keep-all`, 넉넉한 `line-height`, 적절한 `max-width`를 고려한다.

## 동적 UI 기준

- 기도문 카드와 말씀·기도 카드는 단순 그리드보다 3D 스택, 캐러셀, 좌우 이동 UI를 우선 고려한다.
- 좌우 버튼, 키보드 방향키, 모바일 스와이프를 지원하면 좋다.
- 애니메이션은 차분하고 절제되어야 하며, 기도문 읽기를 방해하면 안 된다.
- 검색, 필터, 아카이브 기능은 데이터 기반으로 구현한다.
- 향후 React 전환을 고려해 렌더링 함수와 데이터 구조를 분리한다.

## SEO / AdSense 기준

- 새 HTML 페이지에는 UTF-8, viewport, description, canonical, OG title/description/url/image, title, AdSense account meta, CSS 링크를 포함한다.
- 광고 영역이 본문보다 먼저 보이거나 콘텐츠 경험을 방해하지 않게 한다.
- 광고 클릭 유도 문구는 금지한다.
- 새 페이지를 만들면 `sitemap.xml` 반영 여부를 체크한다.
- 정책 페이지 링크는 유지한다: `privacy.html`, `terms.html`, `disclaimer.html`, `sitemap.xml`.

## 작업 안전 규칙

- 작업 전 `git status --short`로 현재 변경 사항을 확인한다.
- 사용자가 허용한 파일 외에는 수정하지 않는다.
- 기존 콘텐츠, 메뉴 링크, SEO meta, 정책 페이지 링크, AdSense 주석, 유튜브 링크, 문의 링크를 임의로 삭제하거나 변경하지 않는다.
- 디자인 개선이 목적이어도 HTML 구조를 과도하게 갈아엎지 말고 필요한 섹션 단위로 수정한다.
- 새 파일이 필요하면 이유를 설명하고 허용된 경우에만 만든다.
- 문제가 생기면 되돌릴 수 있도록 백업을 먼저 만든다.
- `generate_site.py`는 현재 HTML 인코딩을 깨뜨릴 위험이 있으므로 명시 요청 없이 실행하지 않는다.
- 루트의 `style.css`, `script.js`보다 실제 로드되는 `assets/css/style.css`, `assets/js/script.js`를 우선 확인한다.

## 검증 체크리스트

작업 후 최소한 다음을 확인한다.

- `git status --short`
- `git diff --check`
- JS 파일 변경 시 `node --check`
- 로컬 preview 서버에서 주요 페이지 HTTP 200 확인
- 모바일 폭에서 카드, 버튼, 메뉴가 깨지지 않는지 확인
- 사용자 화면에 `샘플`, `임시`, `준비 중`, `TODO`, `MVP` 같은 표현이 노출되지 않는지 확인
- 유튜브 링크, 문의 링크, 정책 링크가 유지되는지 확인
- 새 페이지가 있으면 `sitemap.xml` 반영 필요 여부를 보고한다.

## 배포 기준

- 배포 전 커밋 범위를 명확히 확인한다.
- 백업 폴더는 일반적으로 배포 커밋에 포함하지 않는다.
- 사용자가 preview 검토 후 배포를 요청하면, 검증을 다시 수행하고 의도한 파일만 커밋한다.
- `git push` 후 원격 브랜치와 커밋 해시를 보고한다.
